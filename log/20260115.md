> [!info] Branch Strategy
>1. Git Flow
>	- main: 배포 코드, develop: 다음 릴리즈 개발, feature: 새 기능 개발, release: 릴리즈 준비, hotfix: 긴급 버그 수정
>	- version이 명시적으로 관리 되어져야 할 때만 사용
>	- 복잡한 branch관리 규약과 branch가 너무 오래 유지된다는 문제점 때문에 비추
> 2. Github Flow
>	- main: 항상 배포 가능한 상태, feature: 기능 개발 후 main에 merge
>	- main에서 branch 생성 -> 작업 후 커밋 -> Pull Request -> 코드 리뷰 -> main에 merge
>	- 이해하기 쉬운 단순한 구조로 배포 사이클이 빠름
>	- feature branch가 너무 오래 유지될 수 있고, 배포와 릴리즈를 분리하기 어렵다는 단점이 있음
> 3.  TBD
>	- 단일 branch로 작업하거나 branch 수명을 매우 짧게 2~3일로 유지
>	- merge X -> rebase
>	- CI/CD가 필수

> [!example] Commit Convention
> type(scope): description
> [Body]
> [Footer]
> `ex) feat(api): login api`
> > [!abstract] Type 종류
> > `feat`: 새로운 기능 추가
> > `fix`: 버그 수정
> > `docs`: 문서 수정(README 등)
> > `style`: 코드 포멧팅, 세미콜론 등
> > `refactor`: 코드 리팩터링
> > `test`: 테스트 코드 추가/수정
> > `chore`: 빌드, 패키지 매니저 설정
> > `ci`: CI/CD 설정 변경

> [!faq] Pull Request?
> 내 브랜치의 변경사항을 다른 브랜치(주로 main)에 병합해달라는 요 청
> > [!abstract] 좋은 PR 4요소
> >  1. Background : 어떤 작업이 포함?
> > 2. Changes : 주요 수정 사항
> > 3. Test Guide : 테스트 방법 안내
> > 4. Important : 리뷰어에게 전달하거나 논의할 내용
> 
> > [!abstract] PR의 종류
